How to debug the 9pi kernel under QEMU
=======================================
(notes from the session where Claude diagnosed why 9pi booted on a real
raspberry pi1 but hung under qemu; kept here as a reusable playbook)

Basic invocation
----------------
QEMU's raspi1ap machine is the same SoC as the pi1 (BCM2835, ARM1176).
There is no VideoCore firmware in QEMU, so three things differ from a real
SD-card boot:

 - -kernel loads raw images at 0x10000 (generic ARM convention) but 9pi is
   linked -H6 -T0x80008000, i.e. physical 0x8000.  Use the generic loader
   to force the address and entry point:
     -device loader,file=kernel/COMPILE/9/bcm/9pi,addr=0x8000,cpu-num=0,force-raw=on
 - the SD card must be a power-of-2 size:
     cp dosdisk.img /tmp/sd.img && qemu-img resize -f raw /tmp/sd.img 1G
 - serial0 is the PL011 (uart0); the mini uart that uartmini.c drives is
   serial1.  So: -serial null -serial mon:stdio

Full command:
  qemu-system-arm -M raspi1ap \
    -device loader,file=kernel/COMPILE/9/bcm/9pi,addr=0x8000,cpu-num=0,force-raw=on \
    -drive file=/tmp/sd.img,if=sd,format=raw \
    -serial null -serial mon:stdio

Caveat: no network is possible; the pi1's ethernet is a USB LAN9512 chip
that QEMU does not model.

'-device usb-kbd -device usb-mouse' work: they enumerate behind QEMU's
emulated LAN9514 hub (like a real pi1) in ~5-6s, the keyboard types cleanly
and the mouse tracks. They originally wedged the boot in an FIQ storm, then
enumerated slowly with a runaway key-repeat bug; both are fixed in the dwcotg
driver -- see the FIQ storm section below and the USB-HID section at the end
(the key fix is disabling split transactions under emulation).

Observing a headless guest
--------------------------
The console output goes to the emulated framebuffer, not the serial port,
so a "silent" guest may actually be printing happily.  Run headless with
the human monitor on stdio and feed it commands through a subshell:

  (sleep 30; echo 'screendump /tmp/s.ppm'; sleep 1; echo 'quit') | \
  qemu-system-arm -M raspi1ap -display none ... -monitor stdio

  convert /tmp/s.ppm /tmp/s.png     # then look at the image

Taking screendumps at increasing times (30s, 90s, 180s) turns the
framebuffer into a crude progress log.

Where is the CPU? (info registers)
----------------------------------
'info registers' in the monitor gives the PC and mode.  Useful facts for
reading the output on this kernel:

 - R15 = PC.  0x80xxxxxx means the MMU is on and we are in kernel virtual
   space (KZERO), i.e. start.s and mmu setup already worked.
 - R10 always holds the Cpu/Mach pointer (0x80002000), R9 holds `up`
   (nil until userinit) -- the 5c compiler reserves these globally.
 - PSR bit 7 (0x80) set = IRQs masked.  '--C- A S svc32' at the end shows
   the mode; if it stays svc32 with IRQs enabled but nothing happens,
   interrupts are not being delivered.

Sampling 'info registers' every couple of seconds is a poor man's
profiler: repeated hits in the same 40-byte window = a spin loop; hits all
over = the system is running, just slow.

Resolving PCs to function names (docs/scripts/aout_syms.py)
-----------------------------------------------------------
The shipped 9pi is -H6 (headerless, no symbol table).  Relink the same
objects without -H6 to get a plan9 a.out with a symtab (the load address
stays the same, no need to boot it -- it is just a symbol map):

  5l -f -o /tmp/9pi.sym -R4096 -T0x80008000 -l <same .5 files and .a libs>

then:  python3 docs/scripts/aout_syms.py /tmp/9pi.sym 80074c18 80085220 ...
  80074c18 -> genrandom +0x14
  80085220 -> memset +0x58

This instantly turned "mystery spin loop at 0x80074c00" into "the
genrandom entropy kproc, which spins by design".

Inspecting memory and MMIO registers (xp)
-----------------------------------------
'xp' in the monitor reads PHYSICAL memory, so subtract KZERO from kernel
virtual addresses.  Two tricks that paid off:

 - dump the Cpu struct twice and diff:
     xp /64wx 0x2000        (Mach/Cpu lives at phys 0x2000)
   If nothing changes between two dumps seconds apart, cpu->ticks is
   frozen: the clock interrupt is dead.  Every driver timeout of the form
   "while(cpu->ticks - now < HZ)" then spins forever.

 - read the device registers the driver is fighting with:
     xp /8wx 0x20003000     (systimer: CS CLO CHI C0 C1 C2 C3)
   Seeing CLO advancing but C3 stuck in the past told us the compare
   register was never re-armed.

Counting exceptions (-d int)
----------------------------
  qemu-system-arm ... -d int -D /tmp/int.log
  grep -c 'Taking exception 5' /tmp/int.log     # IRQs taken

One IRQ in 15 seconds = dead clock.  3000 in 60s = alive (~55 ticks/s).
The log also shows the early banked-mode setup (svc->irq->svc etc. mode
switches from start.s), which confirms the vectors are sane.

Device trace events (--trace)
-----------------------------
QEMU device models have named trace points; list them with:
  qemu-system-arm -M raspi1ap -trace help | grep -i systmr

The smoking gun for the clock bug was:
  --trace 'bcm2835_systmr*' :
    timer write: offset 0x18 data 0xf9e5  -> "timer #3 expiring in 9781 us"   OK
    (fires, kernel acks CS)
    timer write: offset 0x18 data 0x325bd -> "expiring in 4294967231 us"      BUG

i.e. the re-armed compare value had already been passed by the counter, so
the equality-match systimer would only fire again after a full 2^32 us
wrap.  Root cause: under TCG, virtual time = host wall clock, and
interrupt emulation latency (>100us) exceeds the race window that on real
700MHz silicon is nanoseconds.  Hence the two fixes in
kernel/time/arm/clock.c (signed comparisons + re-push c3 until it lands in
the future).

Diagnosing a spontaneous reboot loop (-d cpu_reset)
---------------------------------------------------
After the clock and framebuffer fixes the kernel printed its whole boot
banner and "NNNM memory" but then silently reset, over and over (~285ms
per cycle, banner reprinted 100+ times in 30s).  No panic text, so it was
not a kernel-detected fault.

  qemu-system-arm ... -d cpu_reset -D /tmp/rst.log

logs a full register dump at every CPU reset.  Two things to read off it:

 - is it the guest asking to reset, or QEMU dying?  The initial power-on
   does 3 resets (loader + entry setup); a *reboot* shows up as regular
   reset events much later, interleaved with the mode-switch/IRQ trace.
   Pairs of resets ~6 lines apart = one reboot (the SoC resets twice).

 - the register dump at the reset IS the guilty store's operands, frozen.
   Here R03=7e100000 (= busio 0x7E000000 + 0x100000, the power-manager
   *bus* address) and R02/R06/R11 all had 0x5A... in the top byte -- the
   watchdog Password (0x5A<<24).  So the guest was writing the watchdog
   registers.  R15 (PC) and R14 (LR) name the culprit; feed them to
   aout_syms.py:
     8004b124 -> wdogfeed +0x28
     80064218 -> timerintr +0x1a0
   i.e. timerintr() was calling wdogfeed() every tick.  watchdoglink()
   (registered from the generated pi.c links()) had put wdogfeed on the
   100Hz clock0link list.  On real hw feeding the dog every 10ms keeps its
   5s countdown alive; QEMU's bcm2835 power-manager instead performs the
   reset on the Rstc CfgReset *write itself* (no countdown modelled), so
   every feed = a reset.  Fix: don't register wdogfeed under emulation
   (kernel/init/arm/watchdog.c).  This was the last blocker to reaching a
   shell.

Lesson: a silent reset with no panic is almost always the guest poking a
reset/power register.  -d cpu_reset + aout_syms(PC,LR) finds it in one
run; you do not need to bisect with iprints (though adding a couple of
iprint()s across main()'s tail is a fast way to confirm which phase the
reset lands in -- here, inside init0() right after spllo enabled IRQs).

A FIQ storm at "dossrv: serving" (emulated usb-kbd/usb-mouse)
------------------------------------------------------------
Symptom: with '-device usb-kbd -device usb-mouse' the boot printed its
banner and reached "dossrv: serving #s/dos" and then froze -- no prompt,
forever.  The tempting guess is a dead clock (a stalled tsleep in the
emmc/dma path), but -d int said the opposite:

  grep -c 'Taking exception' int.log      # ~1.9 million in 40s (!)
  grep 'Taking exception 6' int.log | wc  # nearly all of them: FIQ

i.e. an *interrupt storm*, not a stall.  On the pi the dwcotg USB host
controller (usbdwc.c) is wired to FIQ, not IRQ.  The last visible serial
line is "dossrv: serving" only because the boot.rc commands between dossrv
and 'usbd' (mount, bind, ramfs, mkdir, echo>hostowner) print nothing; the
storm actually begins when 'usbd' brings up the dwcotg controller and
QEMU's emulated USB devices start asserting the FIQ line faster than the
driver can quiesce it under emulation.  At ~2M FIQs/s the CPU never makes
forward progress, so it looks hung.

Fix (first cut): the 'run' target dropped usb-kbd/usb-mouse so the boot
reached '%'.  That was a workaround; the storm is now actually fixed in the
driver (see "Making USB HID work" below) and the devices are back.

Lesson mirror of the reset section: a "hang" that is really a *flood* looks
identical from the serial console (nothing happens).  -d int distinguishes
them in one run -- near-zero exceptions = a stall to chase in the sleeping
driver; millions = an interrupt nobody is acking, and 'Taking exception 6'
points straight at FIQ (the USB controller here).

Making USB HID work under QEMU (dwcotg + the sticky Hcintr model)
-----------------------------------------------------------------
Getting usb-kbd/usb-mouse to actually enumerate took a chain of fixes, all
rooted in one QEMU-vs-real-hardware difference.  QEMU's dwc2 model
(hw/usb/hcd-dwc2.c) derives GINTSTS.Hcintr from (HCINT & HCINTMSK) per
channel and only re-evaluates it on a *host-channel* register write; a write
to HAINTMSK is ignored, and GINTSTS.Hcintr is sticky.  Real silicon gates
Hcintr through HAINTMSK combinationally.  So the driver's habit of "mask the
channel in haintmsk and let the sleeping proc clean up HCINT later" leaves
Hcintr stuck asserted under QEMU -> the ~2M/s FIQ storm.  Trace it with
  --trace 'usb_dwc2_*'   (filter out the SOF spam)
which also shows the packet-level story (find_device, handle_packet, status).

The fixes, all in kernel/buses/arm/usbdwc.c unless noted:
 - fiqintr: when a channel completes, clear its hc->hcintmsk (not just the
   haintmsk bit).  That write re-evaluates haint -> lowers Hcintr -> stops
   the storm.  hcint is left intact so the waiting proc still sees the
   result.
 - chandone/chanwait: test hcint against a *saved* wait mask (Ctlr.chanmask),
   not the live hcintmsk that the fiq just cleared, else the completion
   wakeup is lost.  Also make the waits bounded (tsleep 1s) so nothing wedges
   the boot forever.
 - chanio: a 0-length transfer reported Xfercomp is done.  QEMU does not
   decrement hctsiz.Pktcnt for a 0-length packet, so the old "loop until
   Pktcnt changes" re-issued the SET_ADDRESS status stage -- fatal once the
   device had moved to its new address (device_not_found, then a hang).
 - usbd (kernel/buses/user/usb/usbd/usbd.c): the pi's built-in LAN9514 hub
   sits between the root port and the device (QEMU models it too, as
   "QEMU QEMU USB Hub").  QEMU's hub descriptor is 10 bytes for an 8-port hub
   (it omits the deprecated PortPwrCtrlMask high byte); usbd's "descr. too
   small" check demanded 11 and refused to bring the hub up, so nothing
   behind it enumerated.  Relaxed to require only the DeviceRemovable map.
 - sofwait: bounded the split-transaction SOF wait too (same tsleep 1s).

With those, 'cat /dev/usb/ep3.0/ctl' shows the QEMU USB Keyboard/Mouse as a
hid device and the usb/kb driver attaches -- but enumeration took ~25-30s per
device and, once up, the keyboard was unusable: a single tap on 'd' printed a
runaway 'ddddd...' until the next key, then 'eeeee...'.  Both symptoms have
the same root cause, and the same fix.

The split-transaction disable (the real fix)
--------------------------------------------
The pi's keyboard/mouse are full-speed devices sitting behind the high-speed
LAN9514 hub, so real hardware reaches them with *split transactions*: a
start-split on one microframe, a complete-split a few microframes later,
scheduled around the SOF (start-of-frame).  QEMU does not model any of this.
Its dwc2 routes a packet to a device purely by the address in HCCHAR.Devaddr
and forwards it through the emulated hub; the HCSPLT register is stored but
never interpreted (hw/usb/hcd-dwc2.c has no split state machine).  So on real
hardware the split path is mandatory, but under QEMU it is pure overhead:
 - each interrupt poll pays a ~1s sofwait/chanwait timeout (the SOF handshake
   the driver waits for has no counterpart in the model) -> the ~25-30s
   enumeration and seconds-per-poll input lag; and
 - worse, the usb-kbd model ignores SET_IDLE, so it only reports on *change*.
   The driver relies on periodic idle re-reports to recover a key-up edge it
   missed (see the Dwcidle comment in usb/kb).  Under QEMU those re-reports
   never come, so a dropped key-up is never noticed: repeatproc keeps
   auto-repeating the stuck key -> 'ddddd'.

Fix (kernel/buses/arm/usbdwc.c): detect emulation and don't drive the split
state machine at all.  QEMU's dwc2 reports Synopsys id 2.94a (0x4f54294a);
the real BCM2835 core is 2.80a (0x4f54280a).  reset() sets a static
'emulation' flag on the exact QEMU id (any other id, including a different
real board, leaves it 0), and chansetup() then leaves HCSPLT=0 under
emulation even for a device behind a hub.  The transfer completes directly,
like a root-port device: enumeration drops to ~5-6s total and every key/mouse
report is delivered promptly, so taps are single characters and the mouse
tracks.  Real hardware is untouched.

Verify headlessly with the QMP injection recipe in the next section: a single
send-key of 'd','e','f' echoes exactly "def" (not "dddeeefff"), and wiggling
the mouse while 'cat /dev/mouse' runs prints one 'm x y b msec' record per
motion event.

Injecting keyboard and mouse input programmatically (QMP)
---------------------------------------------------------
To test HID without a human at the window, expose the QEMU Machine Protocol on
a unix socket and drive the *same* input subsystem the GUI window feeds:

  qemu-system-arm -M raspi1ap ... -device usb-kbd -device usb-mouse \
    -qmp unix:/tmp/q.sock,server,nowait

Then, after the QMP greeting, send {"execute":"qmp_capabilities"} once and:

 - Keyboard -- 'send-key' presses (and releases) a chord of qcodes:
     {"execute":"send-key",
      "arguments":{"keys":[{"type":"qcode","data":"d"}]}}
   qcodes are QEMU key names: letters "a".."z", "ret", "spc", "delete"
   (interrupt), "minus", "slash", "apostrophe", ...  For a shifted glyph send
   the chord together, e.g. '#' on a US layout is shift+3:
     "keys":[{"type":"qcode","data":"shift"},{"type":"qcode","data":"3"}]

 - Mouse -- 'input-send-event' with a relative-axis event per axis; the
   usb-mouse turns these into HID reports:
     {"execute":"input-send-event",
      "arguments":{"events":[
        {"type":"rel","data":{"axis":"x","value":20}}]}}
     {"execute":"input-send-event",
      "arguments":{"events":[
        {"type":"rel","data":{"axis":"y","value":10}}]}}
   Use "type":"btn" with {"button":"left"|"right"|"middle","down":true|false}
   for clicks, and "type":"abs" (value in the 0..0x7fff range) for absolute
   positioning with a usb-tablet.  To observe the reports in the guest:
     bind -a '#m' /dev      # this minimal boot.rc does not bind the mouse dev
     cat /dev/mouse         # prints 'm <x> <y> <buttons> <msec>' per event

A self-contained Python driver (open the unix socket, do the readline/write
JSON handshake, sleep for enumeration, then send-key / input-send-event) is
docs/scripts/qmp_inject.py.  The protocol is line-oriented JSON and stable
across QEMU versions, so any language works.

Misc
----
 - 'info qtree' shows where the sd-card device is attached (it is on the
   Arasan sdhci bus by default, which is what emmc.c drives -- good).
 - -icount shift=N makes virtual time follow the instruction count, which
   hides latency races but is much slower and its clock warping breaks
   other exact-equality loops; with the clock.c fixes plain TCG is fine.
 - the guest is genuinely slow under TCG: the genrandom kproc burns ~half
   the CPU by design (entropy from timing jitter), and the ARM1176 cycle
   counter does not advance under QEMU, so "cpu0: 0MHz" is cosmetic and
   cpuhz calibration measures zero.

Current status (after clock + fbblank + watchdog + partition fixes)
------------------------------------------------------------------
The kernel boots all the way to an interactive rc shell prompt (%) within a
few seconds and accepts typed commands over the serial console. Just run
'mk run' (see mkfile-target-pi), or by hand:

  qemu-system-arm -M raspi1ap \
    -device loader,file=.../9pi,addr=0x8000,cpu-num=0,force-raw=on \
    -drive file=qemu-sd.img,if=sd,format=raw -serial null -serial mon:stdio \
    -display none

The dossrv "read: 0" story (fixed, and it was NOT a QEMU bug)
------------------------------------------------------------
For a while boot.rc reached a shell but printed, for files past ~100MB:
  dev 3 sector 215271, read: 0, should be 4608   (from dossrv devio.c)
  bind: /rc/bin: '/rc' I/O error
and once the image filled up this broke mount/bind before rc even started.

This was NOT emmc/sdhci: it happened on real hardware too. 'read: 0' with no
kernel 'emmcio:' error means devsd.c sdbio() returned 0 at a *partition*
boundary (bno >= pp->end), i.e. dossrv read past the end of its own
partition. The dos partition was 204801 sectors (100MB) but 'mk disk' runs
mkfs.vfat on the full 512MB image, so the FAT (totsec 1048572) overflowed its
partition; any file placed beyond 100MB was unreachable. Diagnosed purely
statically by parsing the MBR table and the FAT boot sector of dosdisk.img
(no boot needed). Fix: make the partition span the whole 512MB --
arm/boot.rc now hardcodes 'part dos 1 1048577' (like 386/boot.rc) instead of
'fdisk -p' reading the stale MBR table, and MISC/pc/bootsector's table was
corrected to match. Keep these in sync with IMGSIZE in mkfile-host-Linux.
