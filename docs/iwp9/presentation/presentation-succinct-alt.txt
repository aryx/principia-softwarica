FRAME 3: THE EDUCATION GAP
================================================================

"What happens when you type ls in a terminal window?"

Keith Adams — colleague at Facebook, later Chief Architect
  at Slack — used this as his interview question.

Simple question. The answer involves:
  the shell, the C library, the kernel,
  the graphics stack, the windowing system.
A non-trivial amount of code. Most engineers can't answer.

"Full stack developer" = React + Node + cloud
The REAL full stack = compiler, linker, kernel, syscalls

CS curricula shifted to Python, JavaScript, frameworks
Students graduate never seeing how a compiler or OS works
The gap between what's taught and what runs underneath: growing

Mental barrier: engineers don't improve their tools
  because nobody ever showed them the code.
  They think it's impossible. It's not.

FRAME 5: BUT THE STACK IS ENORMOUS
================================================================

Why don't engineers understand the full stack?

  Linux kernel    ~40M LOC  (core ~2M)
  gcc             ~15M LOC
  glibc           ~1.5M LOC
  bash            ~180K LOC
  Xorg server     ~500K LOC (ecosystem: millions)
  xterm alone     ~88K LOC
  # sources:
  # Linux 40M: linuxtoday.com Jan 2025, core ~5%
  # gcc 15M: phoronix.com 2014
  # glibc 1.5M: cloc reports (C+headers)
  # bash 180K: cloc on bash-5.x
  # Xorg 500K: wikipedia X.Org_Server
  # xterm 88K: author's own count

Each component individually: more code than ALL of Plan 9


================================================================
FRAME 6: PLAN 9 — SAME SERVICES, 100x LESS CODE
================================================================

Plan 9 (Thompson, Ritchie, Pike et al.)

Whole OS: ~183,000 LOC — same core services

rio = 8,800 LOC    vs  Xorg = millions
rio terminal < 1K  vs  xterm = 88K  (80x!)

"What happens when you type ls?" — even the OUTPUT is simpler:
  Linux: tty, pty, terminal emulator, VT100 escape sequences...
    a maze of historical layers few people fully understand
  Plan 9: program writes to /dev/cons,
    rio virtualizes /dev/cons per window via namespaces.
    That's it.

The ideal teaching OS must be:
  Open source, Small, Real (not toy!),
  Complete, Coherent, Self-hosting
  Plan 9 is the ONLY OS that checks ALL boxes.
  # Minix: just kernel, needs X Window
  # xv6: no graphics, no networking
  # Oberon: forces Oberon language only
  # TempleOS: no networking
  # GNU/Linux: too big, incoherent patchwork

FRAME 12: WHY BOOKS?
================================================================

"Are books still the best medium in 2026?"

Books have been teaching for ~2000 years (codex form).
  ~580 years in print. Survived radio, TV, internet, MOOCs.
  They will be here for a few hundred more.

The alternatives fall short for large codebases:
  IDE: compiler order, no narrative, everything at once
  Videos: can't reference, can't search, can't study at your pace
  AI explanation: no coherent narrative, no curated ordering

Books = curated narrative + deep focus + completeness
  Knuth's TAOCP, Feynman's Lectures, Euclid's Elements

Syncweb brings IDE features INTO the book:
  Cross-references with hyperlinks = "go to definition"
  Evince inline thumbnails = hover preview in VS Code
  Narrative of a book + navigation of an IDE

FRAME 18: LIVE DEMO
================================================================

Demo: Modify a Syscall, Rebuild, Boot

BEFORE:
  $ strace ls --> open = syscall 6

CHANGE:
  $ vi lib_core/libc/9syscall/sys.h
  #define OPEN 6  -->  #define OPEN 42

REBUILD:
  $ time mk && mk install && mk kernel
  < 10 seconds

BOOT + VERIFY:
  $ qemu-system-i386 -kernel 9qemu -hda dosdisk.img
  $ strace ls --> open = syscall 42

"Rebuilt the entire OS in under a minute. Try that with Linux."

That ls we just traced? It went through the shell, libc,
  the kernel, the graphics stack, the windowing system.
  Every layer — explained in a Principia book.
