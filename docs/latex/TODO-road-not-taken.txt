[Kernel]
- Multics rings: 8 protection rings with fine-grained access control.
  Unix collapsed them to 2 (kernel/user). The intermediate rings could
  have given us hardware-enforced privilege separation for libraries,
  drivers, and services — without the IPC cost of microkernels.

- Persistent operating systems (Grasshopper, KeyKOS): the entire OS
  state survives reboot. No "save" operation, no "startup" sequence.
  Killed by the difficulty of garbage-collecting a persistent heap.
  But the idea persists in databases and Smalltalk images.

- Singularity / Midori (Microsoft Research): type-safe OS written in
  managed code. No hardware memory protection needed because the
  language guarantees safety. Process isolation through software
  verification. Cancelled after 7 years of research. The ideas
  survived partly in Rust's safety model.

- Exokernel (MIT, Engler 1995): give applications direct access to
  hardware, let each app bring its own OS abstractions. Radical idea
  that was too hard to use in practice but the concept reappears in
  unikernels (MirageOS, IncludeOS) and library OSes.

[Machine]
- Itanium / VLIW: move instruction scheduling from hardware (OoO) to
  the compiler (EPIC). Theoretically cleaner — no speculation hardware,
  no power-hungry reorder buffers. But compilers couldn't do it well
  enough for general code. Intel spent billions; x86-64 won instead.
  The idea lives on in DSPs and GPUs where workloads are regular enough.

- Tagged architectures (Burroughs B5000, LISP machines): hardware knows
  the TYPE of every word in memory. No type confusion, no buffer
  overflows by construction. Killed by the overhead of tags and the
  dominance of C's untyped memory model. CHERI (Cambridge) is reviving
  a constrained version with capability tags.

- Dataflow architectures (Manchester, MIT): instructions fire when
  their operands are ready, no program counter. Massive natural
  parallelism. Never solved the memory latency problem. But GPUs
  and FPGAs are essentially dataflow machines with different marketing.

- Stack machines (Forth, JVM, WASM bytecode): simple, compact, easy to
  generate code for. Register machines won for performance (fewer memory
  accesses). But stack-machine bytecodes keep reappearing (JVM, CLR,
  WASM) because they're great as portable intermediate representations.

[Compiler]
- Whole-program optimization: see everything, optimize everything. Early
  Fortran did this. C's separate compilation model made it impractical
  for decades. LTO is a partial comeback, but true whole-program
  optimization (like MLton for Standard ML) remains a niche approach
  despite producing dramatically better code.

- Proof-carrying code (Necula 1997): the compiler emits a proof of
  safety alongside the code; the consumer verifies the proof instead
  of trusting the compiler. Elegant but too expensive in practice.
  WebAssembly's validation is a simplified version of the idea.

- Dependent types (Coq, Agda, Idris): types that depend on values,
  so you can express "this array has exactly n elements" or "this
  function returns a sorted list." Too hard for mainstream programmers
  so far, but the ideas leak into Rust (const generics), TypeScript
  (template literal types), and Liquid Haskell.

[Shell]
- Powershell's structured objects: pipes carry typed objects, not text.
  No more parsing awk fields. Elegant but tied to Windows/.NET, and
  the Unix world rejected it. Nushell (2019) is trying again. The
  idea that pipes should carry structured data has genuine merit that
  Unix's text-stream model sacrificed for simplicity.

- Plan 9 rc's simplicity: 2K lines implementing a complete shell.
  The Unix world chose bash (140K lines) instead. The features rc
  omitted (arrays, associative arrays, arithmetic, built-in regex)
  are arguably better provided by external tools, but the market
  disagreed.

[Editors]
- Structural editing (Smalltalk, Cornell Program Synthesizer, Paredit):
  edit the AST, not the text. No syntax errors possible. But
  programmers preferred text editing — the flexibility of manipulating
  characters freely was worth the occasional syntax error. Tree-sitter
  is bringing some structural awareness back, but true structural
  editing remains a road not taken.

- Oberon's integrated system: editor, compiler, OS all one system.
  Click on an error and jump to the source. Click on a function name
  and execute it. No separate terminal, no separate build system.
  Smalltalk had this too. Modern IDEs approximate it but never achieved
  the integration of a system designed as one piece.

[Linker]
- Smart linking / dead code elimination at link time: the linker sees
  all code and can remove unused functions. Plan 9's linker does this;
  Go inherited it. But most Unix linkers link at object-file granularity,
  not function granularity, pulling in entire .o files even for one
  symbol. LTO and -ffunction-sections/-gc-sections are workarounds for
  a design decision made in the 1970s.

[Utilities]
- Semantic Unix tools: tools that understand their input's structure,
  not just bytes. diff that understands syntax trees (not lines),
  grep that understands scopes (not regexes), sort that understands
  data types (not byte order). Plan 9's plumber and structural
  regular expressions (sam) hinted at this. Mostly unrealized.

[Lib_core]
- Region-based memory management (Tofte-Talpin 1997, Cyclone): allocate
  in a region, free the entire region at once. No fragmentation, no
  use-after-free within a region, deterministic deallocation. Theoretically
  beautiful, but requires the programmer to think about lifetimes
  upfront. Rust's borrow checker is a distant relative.

- Software transactional memory (STM): instead of locks, wrap shared
  memory access in transactions that retry on conflict. Haskell's STM
  (2005) works beautifully. But hardware/C/C++ implementations never
  delivered the performance promise (Intel TSX was buggy, disabled).
  The idea is sound; the implementation is waiting for the right hardware.

[Lib_networking]
- Active networking (1990s): routers execute code on packets in transit.
  Too dangerous and too slow at the time. eBPF in the kernel is a
  constrained, safe version of the same idea — and it's succeeding
  precisely because it added the safety guarantees the original vision
  lacked.

- OSI's session/presentation layers: the idea that the network stack
  should handle data format negotiation and session management. TCP/IP
  punted this to applications. Now every application reinvents session
  management (cookies, tokens, TLS renegotiation) and data format
  (JSON, protobuf, msgpack). Maybe OSI was right about the layers,
  just wrong about the implementation.

[Windows]
- Display PostScript (NeXT, 1988): the display server speaks PostScript,
  so rendering is resolution-independent and WYSIWYG by construction.
  Adobe's licensing killed it. PDF is its spiritual successor but
  not as a display server. macOS Quartz (2001) finally delivered
  resolution-independent rendering, but via a different path.

- Network-transparent GUIs (X11's original vision): any application
  can display on any screen over the network. Wayland dropped this.
  Modern remote desktops (RDP, VNC) work at the pixel level instead.
  The application-level semantic approach had genuine advantages
  (lower bandwidth, resolution independence) that we lost.

[Filesystem]
- Object storage in the filesystem: instead of bytes in files, store
  typed objects with metadata. WinFS (Microsoft, cancelled 2006) tried
  this. BeOS had queryable metadata on files. Plan 9's plumber understood
  file types. The idea keeps getting reinvented (Spotlight, Tracker)
  but never as the primary filesystem abstraction.

- Log-structured filesystems (Rosenblum & Ousterhout 1991): write
  everything sequentially like a log. Perfect for flash storage, which
  arrived 15 years later. LFS was ahead of its time — now F2FS and
  flash translation layers use the same ideas internally.
