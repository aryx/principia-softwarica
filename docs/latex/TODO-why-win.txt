[Machine]
- x86 over everything: not technically best, but IBM PC gave it installed base;
  backward compatibility kept it alive decade after decade; AMD64 extension
  kept it going when Intel tried to replace it with Itanium. The "wintel"
  lock-in is one of the strongest network effects in computing history.

- ARM over MIPS/SPARC/PA-RISC in embedded: low power consumption (the "A"
  originally stood for Acorn); licensing model let anyone build a chip;
  phone revolution (iPhone 2007) gave it billions of units of volume.

- Little-endian over big-endian: Intel's x86 was little-endian, and x86 won;
  ARM switched from big-endian default to little-endian; network byte order
  is still big-endian (historical artifact) but the host won.

[Kernel]
- Unix over Multics/VMS/OS/360: simplicity (Thompson/Ritchie could build it
  on a PDP-7), free academic distribution (BSD), portability via C. The
  antitrust consent decree forcing AT&T to license it cheaply was an
  accidental catalyst.

- Linux over other free Unixes (386BSD, Hurd, Minix): timing (available when
  386BSD had legal issues), pragmatic monolithic design (worked NOW vs Hurd's
  microkernel perfectionism), Torvalds's welcoming community + early adoption
  of the internet for collaboration.

- Preemptive multitasking over cooperative: one misbehaving app can't freeze
  the system. Classic Mac OS cooperative model meant a spinning beach ball
  locked everything; Windows 3.1 same. NT and Unix always preemptive.

- Flat memory model over segmentation: simpler for compilers, simpler for
  programmers, no "near/far pointer" confusion. 80386 offered both; everyone
  chose flat. AMD64 removed segmentation entirely (mostly).

[Compiler]
- C over Pascal/Modula-2 for systems programming: Unix was written in C,
  so every Unix system shipped a C compiler. Separate compilation + linker
  model scaled better than Pascal's whole-program compilation. The
  preprocessor was ugly but enabled portability (#ifdef). K&R book was
  excellent. Pascal had no standard way to call OS services.

- LLVM over GCC as compiler infrastructure: permissive license (BSD vs GPL)
  let Apple/Google/etc. build proprietary toolchains; clean modular IR
  designed from scratch vs GCC's evolved internals; Clang's much better
  error messages proved the architecture's value.

- GCC over proprietary compilers (Sun cc, IBM xlc, SGI MIPSpro): free (as in
  beer AND freedom), ran everywhere, accumulated decades of optimization work.
  Once GCC was "good enough," paying for a compiler became hard to justify.

[Linker]
- Dynamic linking over static (1990s-2000s): shared libraries saved disk and
  RAM when both were expensive; security patches could update one .so instead
  of relinking every program. These advantages were real and dominant for
  20 years (before the static linking comeback).

[Shell]
- bash over other shells (csh, ksh, tcsh): GNU/Linux default. That's almost
  the entire explanation. ksh was technically comparable but proprietary until
  2000. csh had broken scripting ("Csh programming considered harmful").
  zsh was better but came later. Default wins.

- Unix pipes over Multics's approach: simplicity of byte-stream interface.
  No structured records to negotiate, no type system to agree on. Any program
  that reads stdin and writes stdout can participate. The composability
  this enables is worth more than type safety in practice.

[Utilities]
- grep/awk/sed survival over "better" replacements: composability + muscle
  memory + ubiquity. Perl was supposed to replace them (and partly did for
  scripts) but for one-liners at the terminal nothing beats grep | awk.
  The tools are so deeply embedded in build systems and deployment scripts
  that replacing them would be like replacing ASCII.

- make over predecessors (nothing): make (Feldman 1976) was the first build
  tool; it won by being first. Its survival over successors (Ant, Maven,
  CMake, Bazel) is a different story: simplicity, universality, and the
  fact that POSIX standardized it.

[Editors]
- vi/vim survival: modal editing is more efficient for experienced users
  (fewer keystrokes for common operations). Runs everywhere (every Unix
  has vi). Extremely fast on large files. But really: ubiquity + muscle
  memory + decades of accumulated plugins.

- VS Code over Atom/Sublime/others: Electron done right (fast enough),
  Microsoft's resources for polish, extension marketplace with critical mass,
  free. TypeScript gave it best-in-class JavaScript/TS support which was the
  growth language of the 2010s.

[Debugger]
- printf-debugging over formal debuggers: zero setup, works in any language,
  works across network boundaries, works in production, doesn't change timing
  much (vs breakpoints that stop the world). Every survey shows most
  developers use printf most of the time despite having GDB/LLDB available.

- GDB over other Unix debuggers (dbx, adb): GNU project's momentum + Linux
  default. Same pattern as bash — the GNU/Linux default wins.

[Profiler]
- Sampling over instrumentation for production profiling: low overhead
  (typically <5%) vs instrumentation's 10-50x slowdown. perf, DTrace,
  and eBPF all sample. Instrumentation is too expensive for always-on
  production use.

[Network]
- HTTP/REST over SOAP/XML-RPC/CORBA: simplicity. A URL + GET/POST is
  something anyone can test with curl. SOAP required tooling, WSDL files,
  code generation. REST won because the barrier to entry was near zero.

- JSON over XML for APIs: simpler to read, simpler to parse, native to
  JavaScript (the browser language). XML is more expressive (schemas,
  namespaces, XSLT) but that expressiveness became a liability.

[Windows]
- Overlapping windows over tiling: direct manipulation metaphor — drag a
  window feels like moving paper on a desk. The Xerox Star (1981) and
  Mac (1984) made this the standard. Tiling is more efficient for
  power users but overlapping is more intuitive for newcomers.

[Lib_core]
- malloc/free over region-based allocation for general use: flexibility.
  Regions require knowing allocation lifetimes at design time; malloc
  handles arbitrary lifetimes. The cost is fragmentation and use-after-free
  bugs, but the generality won.

- IEEE 754 floating point: ended the chaos of incompatible formats (IBM,
  VAX, Cray all different). Kahan's design gave deterministic rounding,
  NaN propagation, and gradual underflow. Once Intel put it in the 8087
  coprocessor (1980) and then integrated it into every x86 CPU, the
  installed base made alternatives impossible.

[Lib_gui]
- Callback-based event handling over subclassing: simpler to use. Early
  toolkits (Motif, early Java AWT) required subclassing a widget to
  handle events. Callbacks (Tk, GTK signals, Qt signals/slots, JavaScript
  addEventListener) let you attach behavior without creating a class
  hierarchy. The functional style won.

[Filesystem]
- Hierarchical filesystems over flat: directories give namespace structure
  that scales. Early systems (CP/M, RT-11) were flat; once you had more
  than ~100 files it became unmanageable. Multics introduced the tree
  (1965), Unix inherited it, and every system since has kept it.

- Journaling filesystems over fsck: crash recovery in seconds instead of
  minutes/hours. ext3 (2001) brought journaling to Linux default; NTFS
  had it since 1993. The cost (slight write overhead) is invisible
  compared to the benefit of not losing a weekend to fsck.
