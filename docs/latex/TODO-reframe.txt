[Kernel]
- Kernel as the first reactive program: the kernel sits idle until
  an event (syscall, interrupt, timer) wakes it. This is the same
  event-driven model that Node.js and nginx made fashionable — the
  kernel was doing it in 1969.

- Kernel as the first multicore program: SMP kernels solved the
  shared-state concurrency problem decades before application
  programmers had to worry about it. Every locking pattern in your
  Go/Rust code was invented for kernel use.

- Syscall interface as API design: the kernel's syscall table is
  the oldest and most stable API in computing. read/write/open/close
  have been stable for 50+ years. What makes an API survive that
  long? The kernel is a masterclass in interface design.

- Per-process namespace as dependency injection: Plan 9's bind/mount
  lets you swap out /net, /dev, /srv per process. This is the same
  idea as dependency injection in software engineering — but at the
  OS level, with files as the universal interface.

[Compiler]
- Compiler as the most careful code reader: it traces every variable,
  checks every type, detects dead code — it reads your code more
  carefully than any human reviewer. (Already partially tagged as
  %why-study: — could add %reframe: alongside.)

- Type checker as theorem prover: every type annotation is a
  proposition; every well-typed program is a proof. The
  Curry-Howard correspondence means the compiler is doing
  mathematical reasoning, even for mundane C code.

- Optimization passes as program transformation: the compiler
  doesn't just "translate" — it rewrites your program into a
  different, equivalent program that happens to be faster. It's
  an automated refactoring engine.

- Register allocator as graph coloring: a CS-theory result
  (NP-completeness of graph coloring) directly determines how
  your code runs. The compiler is solving a combinatorial
  optimization problem millions of times a day.

[Shell]
- Shell as glue language: the shell's real job isn't running
  commands — it's connecting programs. Pipes, redirections, and
  variables are the syntax of composition. The shell is a
  component wiring language.

- Shell pipeline as functional composition: ls | grep | sort | head
  is f(g(h(x))). Each program is a pure function from stdin to
  stdout. The shell is a functional programming environment where
  the functions happen to be executables.

- Shell as the original REPL: read a command, evaluate it, print
  the result, loop. Every language REPL (Python, Lisp, Node)
  follows the pattern the shell established.

[Linker]
- Linker as the program that completes the compiler's work: the
  compiler produces fragments with holes (unresolved symbols);
  the linker fills the holes. Compilation is not one program's
  job — it's a pipeline of transformations across multiple tools.

- Symbol table as a contract: the .o file's symbol table is a
  promise ("I provide foo, I need bar"). The linker is a contract
  verifier that checks all promises are fulfilled.

- Relocation as late binding: the linker patches addresses at the
  last possible moment. This is the same principle as late binding
  in OOP — defer decisions until you have enough information.

[Debugger]
- Debugger as program-reading tool, not bug-finding tool: the
  debugger's primary value is making execution visible. It's a
  microscope for programs — you learn more by watching a program
  run than by reading its source.

- ptrace as the debugger's syscall: just as programs use syscalls
  to talk to the kernel, debuggers use ptrace to talk to other
  programs. The debugger is a meta-program that controls other
  programs, just as the kernel is a meta-program that runs them.

- Breakpoint as self-modifying code: the debugger overwrites an
  instruction with a trap. This is sanctioned self-modifying
  code — the oldest trick in computing, formalized into a tool.

[Profiler]
- Profiler as statistical sampler: a profiler doesn't measure
  execution — it samples it, like a political poll. Understanding
  sampling theory (confidence intervals, bias, aliasing) is
  understanding what the profiler can and cannot tell you.

- Flamegraph as a map: a flamegraph is a geographic map of your
  program's time. Width is time, depth is call stack. You read
  it like a topographic map — look for the wide plateaus (hot
  functions), not the tall spikes (deep but fast call chains).

[Machine]
- CPU as interpreter: the fetch-decode-execute loop IS an
  interpreter loop. The CPU interprets machine code the same way
  Python interprets bytecode — just faster, in silicon. An
  emulator makes this explicit.

- Memory as a giant array: at the hardware level, memory is just
  addressable bytes. Every abstraction above it (stack, heap,
  virtual memory, cache) is software/hardware creating the
  illusion of structure on top of a flat array.

[Assembler]
- Assembler as the simplest compiler: it does exactly what a
  compiler does (translate a language to machine code) but with
  a 1:1 mapping between source and output. Understanding the
  assembler is understanding the compiler's back end, stripped
  of optimization.

- Object file as a contract: an .o file says "here is code and
  data, here are the symbols I define, here are the symbols I
  need, here is how to patch addresses." It's a serialized
  promise between the assembler and the linker.

[Utilities]
- Unix pipeline as dataflow programming: data flows through a
  graph of processes connected by pipes. This is the same model
  as Apache Beam, Spark, and TensorFlow — the Unix shell just
  got there 40 years earlier.

- Regular expression as a tiny virtual machine: Thompson's
  construction compiles a regex to an NFA, which is essentially
  a tiny state machine. grep is running a virtual machine on
  every line of your file.

- Each utility as a microservice: grep, sort, awk are independent
  services that communicate via a standard protocol (text on
  stdin/stdout). The Unix philosophy IS microservice architecture,
  predating the term by decades.

[Lib_core]
- malloc as an OS within the process: malloc manages a heap the
  way the kernel manages physical memory — free lists, splitting,
  coalescing, fragmentation. It's a memory manager inside a
  memory manager.

- printf as a tiny interpreter: the format string is a program;
  printf interprets it, dispatching on %d, %s, %x. It's a
  domain-specific language embedded in C via varargs.

[Lib_networking]
- Network stack as nested virtual machines: each layer (Ethernet,
  IP, TCP, HTTP) is a virtual machine that runs on top of the
  one below. A packet is unwrapped layer by layer, like Russian
  dolls.

- 9P as the network IS the file system: Plan 9's 9P makes remote
  files indistinguishable from local ones. The network doesn't
  connect to the filesystem — the network IS the filesystem.
  This is a deeper integration than NFS ever achieved.

[Lib_graphics]
- Bitblt as the universal graphics primitive: every pixel
  operation (draw, copy, scroll, XOR cursor) is a special case
  of bitblt with different source, mask, and operation. One
  function to rule them all.

[Filesystem]
- Filesystem as a naming system: the filesystem's deepest job
  isn't storing bytes — it's mapping names to data. The
  hierarchy, permissions, and metadata are all about naming and
  finding, not about storage. Plan 9 takes this to its logical
  conclusion: everything is a name in a filesystem, even
  devices and network connections.

[Editors]
- Text editor as a database: the editor manages a data structure
  (the buffer) with CRUD operations (insert, delete, replace,
  read), transactions (undo/redo), and concurrent access (multiple
  views). Gap buffers and ropes are specialized database storage
  engines.

[Formatters]
- troff pipeline as compiler pipeline: preprocessors (tbl, eqn,
  pic) are analogous to compiler front-end passes; troff itself
  is the code generator; the output driver is the target backend.
  Document formatting IS compilation, with a different input
  language.
